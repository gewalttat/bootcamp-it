ОБЩЕЕ ОПИСАНИЕ
════════════════════════════════════════════════════════════════════════════════

WB Junior Clone это полнофункциональное веб-приложение e-commerce, упрощенный клон маркетплейса 
Wildberries. Проект демонстрирует как устроены современные интернет-магазины: полный цикл от 
просмотра каталога товаров до добавления их в корзину, плюс админ-панель для управления товарами. 

Основные функции приложения:
- Просмотр каталога товаров с изображениями и описаниями
- Поиск товаров по названию и бренду
- Фильтрация товаров по категориям (Электроника, Одежда и обувь, и т.д.)
- Сортировка товаров (по цене, рейтингу, скидке)
- Добавление товаров в корзину с сохранением между сеансами браузера
- Админ-панель для добавления и удаления товаров
- Правильная архитектура - frontend отделен от backend

Проект очень хороший для учебы потому что показывает реальную структуру веб-приложения. Это не просто 
страница с кнопками - это настоящее приложение со своей логикой, серверной частью и базой данных.


ТЕХНОЛОГИЧЕСКИЙ СТЕК
════════════════════════════════════════════════════════════════════════════════

FRONTEND (клиентская часть - в браузере):

1. HTML5 - семантическая разметка, правильная структура страницы. Используется для создания 
   основной структуры интерфейса (заголовки, товары, модальные окна, корзина).

2. CSS3 - стилизация и адаптивный дизайн. Применяется Flexbox и Grid для правильной раскладки.
   Страница работает на мобильных и десктопных устройствах.

3. Vanilla JavaScript ES6+ - чистый JavaScript без фреймворков типа React. Это очень важно для 
   обучения потому что показывает как JavaScript работает на низком уровне. Используются:
   - Работа с DOM (получение элементов, изменение содержимого)
   - Событийная модель (обработка клика, отправка форм)
   - Fetch API для HTTP запросов к серверу
   - Promises и async/await для асинхронного кода
   - localStorage для сохранения данных в браузере

BACKEND (серверная часть):

1. Node.js - JavaScript runtime на сервере. Это позволяет использовать JavaScript как на фронтенде, 
   так и на бэкенде. Node.js работает асинхронно и может обрабатывать много запросов одновременно.

2. Express.js - легкий веб-фреймворк для Node.js. Библиотека которая упрощает создание HTTP сервера.
   Используется для создания REST API, обработки запросов и маршрутизации.

3. CORS (Cross-Origin Resource Sharing) - middleware который разрешает браузеру делать запросы 
   с одного домена на другой. Нужен потому что фронтенд и бэкенд работают на разных портах 
   (браузер на локальном пути, сервер на localhost:3000).

4. fs модуль (File System) - встроенный модуль Node.js для работы с файлами. Используется для 
   чтения и записи базы данных (db.json файл).

5. JSON - простой текстовый формат для хранения данных. Вместо SQL базы (PostgreSQL, MySQL) 
   используется обычный JSON файл. Это проще для учебы и демонстрации, но в реальных приложениях 
   используют настоящие базы данных.

ВСЕ ОСТАЛЬНОЕ:

1. REST API - архитектурный стиль для организации сервера. Вместо непонятных URL типа 
   /update_product_and_notify_user используются стандартные HTTP методы:
   - GET /api/products - получить данные (безопасно, можно кэшировать)
   - POST /api/products - создать новый ресурс (изменяет состояние)
   - PUT /api/products/:id - обновить ресурс целиком
   - DELETE /api/products/:id - удалить ресурс

2. HTTP протокол - стандартный протокол общения в интернете. Браузер отправляет HTTP запросы,
   сервер отправляет HTTP ответы.

3. npm (Node Package Manager) - менеджер пакетов для Node.js, как pip в Python. Позволяет 
   устанавливать библиотеки и управлять зависимостями проекта.

4. package.json - конфигурационный файл проекта. Хранит список зависимостей, версию проекта, 
   команды для запуска.

5. localStorage - встроенный API браузера для сохранения данных на компьютере пользователя. 
   Данные сохраняются даже после закрытия браузера. Используется для сохранения корзины.

Архитектурный паттерн: Клиент-Серверная модель

Frontend (Клиент):
- Отвечает исключительно за пользовательский интерфейс
- Отправляет запросы на сервер когда нужны данные
- Сохраняет локальные данные в localStorage (например, корзина)
- Выполняет простую валидацию данных перед отправкой
- Обрабатывает события пользователя (клики, ввод текста, отправка форм)

Backend (Сервер):
- Хранит основную базу данных товаров
- Обрабатывает все запросы от клиента
- Выполняет операции чтения/записи данных
- Отвечает за предпринимательскую логику (например, проверка корректности товара)
- Отправляет ответы обратно клиенту в формате JSON

Такое разделение позволяет:
- Легко масштабировать (если много пользователей, просто добавляем серверы)
- Переиспользовать бэкенд (можно сделать мобильное приложение без изменений в коде сервера)
- Отделить ответственность (фронентенд не знает о БД, бэкенд не знает о UI)

МОДУЛЬНАЯ АРХИТЕКТУРА FRONTEND
════════════════════════════════════════════════════════════════════════════════

Весь JavaScript код разбит на 8 независимых модулей. Каждый модуль отвечает за одну конкретную 
задачу. Это называется "Принцип единственной ответственности" (Single Responsibility Principle).

1. api.js - РАБОТА С СЕРВЕРОМ
   Это модуль для коммуникации с бэкенд сервером через HTTP запросы.
   
   Основные функции:
   - loadGoods() - загрузить товары с сервера (GET /api/products)
   - getGoodById(id) - получить один товар по ID
   - getAllGoods() - получить все товары (может быть уже загруженные)
   - getGoodsByCategory(category) - получить товары определенной категории
   - searchGoods(query) - поиск товаров по названию или бренду
   - addProductToServer(product) - отправить новый товар на сервер (POST)
   - deleteProductFromServer(id) - удалить товар с сервера (DELETE)
   
   Внутренне используется Fetch API:
   fetch('http://localhost:3000/api/products')
       .then(response => response.json())
       .then(data => { /* обработка данных */ })
   
   Или современный синтаксис с async/await:
   const data = await fetch(url).then(r => r.json());

2. storage.js - ЛОКАЛЬНОЕ ХРАНИЛИЩЕ
   Этот модуль работает с localStorage браузера для сохранения данных на диске 
   пользователя без отправки на сервер.
   
   Основные функции:
   - saveCart(cartItems) - сохранить корзину в localStorage
   - loadCart() - загрузить корзину из localStorage при открытии приложения
   - clearCart() - полностью очистить корзину
   - saveFavorites(favorites) - сохранить избранное
   - loadFavorites() - загрузить избранное
   
   localStorage работает как простой ключ-значение хранилище:
   localStorage.setItem('cart', JSON.stringify(cartItems));
   const loadedCart = JSON.parse(localStorage.getItem('cart'));
   
   Важно: это браузерное хранилище, данные не синхронизируются между браузерами.

3. cart.js - УПРАВЛЕНИЕ КОРЗИНОЙ
   Модуль который содержит всю логику работы с корзиной.
   
   Основные функции:
   - initCart() - инициализировать корзину при загрузке приложения
   - addToCart(id) - добавить товар в корзину
   - removeFromCart(index) - удалить товар из корзины по индексу
   - clearAllCart() - полностью очистить всю корзину
   - getCartCount() - получить количество товаров в корзине
   - getCartTotal() - получить общую сумму всех товаров
   - getCartItems() - получить массив всех товаров в корзине
   - updateCartCounter() - обновить счетчик товаров в UI (число в тег)
   
   Корзина хранится в памяти JavaScript но также сохраняется в localStorage 
   для персистентности (чтобы не потерялась при обновлении страницы).

4. catalog.js - ОТРИСОВКА ТОВАРОВ
   Модуль отвечает за отображение товаров на странице (через HTML).
   
   Основные функции:
   - renderGoods(data) - вывести товары на страницу (перебирает массив и создает HTML)
   - openModal(id) - открыть модальное окно с подробной информацией о товаре
   - closeModal() - закрыть модальное окно
   - addToCartAndClose(id) - добавить товар в корзину и закрыть модаль
   
   renderGoods берет массив товаров и создает для каждого HTML карточку:
   <div class="goods-item">
       <img src="товар.картинка">
       <h3>товар.название</h3>
       <p>товар.цена</p>
       <button onclick="addToCart(товар.id)">Добавить</button>
   </div>

5. filter.js - ФИЛЬТРАЦИЯ И СОРТИРОВКА
   Модуль для фильтрации и сортировки товаров (без отправки на сервер).
   
   Основные функции:
   - filterCategory(categoryName) - оставить только товары определенной категории
   - sortByPrice(order) - отсортировать по цене (asc = дешевле, desc = дороже)
   - sortByRating() - отсортировать по рейтингу (от лучших к худшим)
   - sortByDiscount() - отсортировать товары с большей скидкой первыми
   - filterByPrice(min, max) - оставить справедливые товары в диапазоне цен
   - filterByRating(minRating) - оставить товары с рейтингом не ниже чем указано
   
   Все фильтры и сортировки выполняются локально на фронтенде через array.filter() 
   и array.sort(), затем перерисовывается каталог через catalog.renderGoods().

6. header.js - ХЕДЕР И НАВИГАЦИЯ
   Модуль для функций хедера (верхняя часть страницы) и основной навигации.
   
   Основные функции:
   - initSearch() - инициализировать поиск (добавить слушатель на input)
   - onSearchInput(query) - выполняется при вводе в поле поиска
   - searchGoods(query) - поиск товаров и перерисовка каталога
   - openCart() - открыть модальное окно с корзиной
   - closeCart() - закрыть модальное окно с корзиной
   - updateCartCounter() - обновить число в иконке корзины
   - checkoutOrder() - оформить заказ (например, очистить корзину)
   
   Хедер содержит логотип, поле поиска и иконку корзины. Когда пользователь 
   вводит в поле поиска - вызывается поиск и перерисовывается каталог.

7. admin.js - УПРАВЛЕНИЕ ТОВАРАМИ (НОВОЕ!)
   Модуль админ-панели для создания и удаления товаров через API.
   
   Основные функции:
   - initAdmin() - инициализировать админ-панель
   - openAddProductModal() - открыть форму добавления товара
   - closeAddProductModal() - закрыть форму
   - addProductToServer(product) - отправить товар на сервер (POST запрос)
   - deleteProductFromServer(id) - удалить товар с сервера (DELETE запрос)
   - validateProduct(product) - проверить что товар имеет все необходимые поля
   - fillRandomImage() - если изображение не указано, генерировать случайную картинку
   
   Админ-панель это обычная HTML форма с полями (название, цена, категория и т.д.). 
   Когда пользователь заполняет форму и нажимает "Добавить товар" - 
   admin.js отправляет данные на сервер через api.js (POST /api/products).

8. main.js - ИНИЦИАЛИЗАЦИЯ ПРИЛОЖЕНИЯ
   Это точка входа приложения. main.js запускается сразу при загрузке index.html.
   
   Основные функции:
   - Загрузить все товары с сервера через api.loadGoods()
   - Инициализировать корзину через cart.initCart()
   - Загрузить сохраненную корзину из localStorage через storage.loadCart()
   - Отрисовать товары на странице через catalog.renderGoods()
   - Инициализировать все обработчики событий и слушатели
   - Проверить什么 на странице загружено (это не требует явного вызова, обв happens automatically)
   
   Типичный порядок инициализации:
   1. Загрузить товары с сервера
   2. Инициализировать хранилище
   3. Отрисовать товары
   4. Иициализировать модули (search, cart, filters, admin)
   5. Приложение готово к использованию


BACKEND: EXPRESS СЕРВЕР (Node.js)
════════════════════════════════════════════════════════════════════════════════

Server.js - это основной файл бэкенда. Он создает Express сервер и определяет 
все API endpoints (URL маршруты) для работы с товарами.

Основные компоненты сервера:

1. Инициализация Express
   const express = require('express');
   const cors = require('cors');
   const app = express();
   const PORT = 3000;

2. Мидлвары (Middleware)
   app.use(cors());          // Разрешаем кроссдоменные запросы (фронтенд на другом порту)
   app.use(express.json());  // Парсим JSON из тела запроса

3. Работа с файлами БД
   // Функция для чтения БД
   const readDB = () => JSON.parse(fs.readFileSync(DB_PATH, 'utf-8'));
   
   // Функция для записи БД
   const writeDB = (data) => fs.writeFileSync(DB_PATH, JSON.stringify(data, null, 2));
   
   // При первом запуске создаем初始 товары если файла нет
   if (!fs.existsSync(DB_PATH)) {
       writeDB([
           { id: 1, title: "iPhone 15", price: 95000 },
           { id: 2, title: "Galaxy S23", price: 80000 }
       ]);
   }

4. REST API Endpoints - четыре операции CRUD:

   GET /api/products
   - Описание: Получить список всех товаров
   - Запрос: никаких параметров
   - Ответ: JSON массив всех товаров
   - Пример: fetch('http://localhost:3000/api/products')
   
   POST /api/products
   - Описание: Добавить новый товар
   - Запрос: JSON в теле (название, цена, категория и т.д.)
   - Ответ: JSON новый созданный товар с ID
   - Пример: fetch('/api/products', {
       method: 'POST',
       headers: { 'Content-Type': 'application/json' },
       body: JSON.stringify({ title: 'iPhone 16', price: 110000 })
     })
   
   PUT /api/products/:id
   - Описание: Обновить существующий товар
   - Параметр :id - это ID товара которого нужно обновить
   - Запрос: JSON с новыми данными товара
   - Ответ: JSON обновленный товар
   - Пример: fetch('/api/products/1', {
       method: 'PUT',
       headers: { 'Content-Type': 'application/json' },
       body: JSON.stringify({ title: 'iPhone 15 Pro', price: 105000 })
     })
   
   DELETE /api/products/:id
   - Описание: Удалить товар по ID
   - Параметр :id - ID удаляемого товара
   - Запрос: пусто
   - Ответ: JSON сообщение об успехе
   - Пример: fetch('/api/products/1', { method: 'DELETE' })

5. Обработка ошибок
   Каждый endpoint оборачивается в try-catch:
   try {
       // обработка запроса
       res.json({ data });
   } catch (err) {
       res.status(500).json({ error: "Ошибка сервера" });
   }
   
   Если что-то идет не так (например, файл БД поврежден) - сервер возвращает 
   ошибку 500 вместо падения.

БАЗА ДАННЫХ (db.json)
════════════════════════════════════════════════════════════════════════════════

Вместо SQL базы данных (PostgreSQL, MySQL) используется простой JSON файл:

[
  {
    "id": 1,
    "category": "Электроника",
    "title": "Смартфон iPhone 15",
    "brand": "Apple",
    "price": 95000,
    "oldPrice": 120000,
    "rating": 4.9,
    "img": "images/12.webp"
  },
  {
    "id": 2,
    "category": "Одежда и обувь",
    "title": "Кроссовки беговые летние",
    "brand": "SportLine",
    "price": 3200,
    "oldPrice": 5500,
    "rating": 4.5,
    "img": "images/12.webp"
  },
  // ... еще товары
]

Структура товара:
- id: уникальный идентификатор (число)
- category: категория товара (строка)
- title: название товара (строка)
- brand: бренд товара (строка)
- price: текущая цена товара (число)
- oldPrice: старая цена (для расчета скидки)
- rating: рейтинг товара (число от 0 до 5)
- img: путь к изображению товара (строка)

Преимущества JSON файла для обучения:
+ Просто понять как работает
+ Не нужно устанавливать БД
+ Можно открыть блокнотом и увидеть данные
+ Быстро разработать и прототипировать

Минусы (почему в продакшене используют SQL):
- Масштабируемость - при миллионах товаров будет медленно
- Синхронизация - если несколько серверов, данные могут разойтись
- ACID гарантии - нет гарантий что при краше сервера данные не потеряются
- Сложные запросы - невозможно сделать JOIN с другими таблицами (нет таблиц)

Сценарий 1: ЗАГРУЗКА ТОВАРОВ при открытии приложения

   app.get('/api/products', (req, res) => {

   fetch('/api/products', {
       method: 'POST',

   })

   app.post('/api/products', (req, res) => {

writeDB() saves to db.json → 
response with new product → api.loadGoods() → 
catalog.renderGoods() → User sees new product


КЛЮЧЕВЫЕ КОНЦЕПЦИИ ПРОГРАММИРОВАНИЯ
════════════════════════════════════════════════════════════════════════════════

1. МОДУЛЬНОСТЬ (Modularity)

Код разбит на отдельные файлы (модули), каждый отвечает за одну функцию.

Преимущества:
- Легче читать - когда ищешь функцию поиска, ты идешь в header.js
- Легче тестировать - можно протестировать cart.js отдельно от filter.js
- Легче переиспользовать - модуль cart.js можно скопировать в другой проект
- Легче масштабировать - добавлять новые модули просто, без конфликтов

Недостаток:
- Нужно помнить какой модуль что делает (но это решается документацией)

Пример плохой модульности (не так):
// Один большой файл на 5000 строк
app.js (contains: renderGoods, addToCart, fetchAPI, sortByPrice, searchGoods, ...)

Пример хорошей модульности (так правильно):
- api.js (только работа с сервером)
- catalog.js (только отрисовка)
- cart.js (только логика корзины)
- filter.js (только фильтрация)


2. REST API И HTTP МЕТОДЫ

REST это не класс или функция - это архитектурный паттерн (стиль) создания веб-сервиса.
Идея в том что используются стандартные HTTP методы для стандартных операций:

GET - получение данных (безопасно, можно повторять много раз без побочных эффектов)
POST - создание нового ресурса (изменяет состояние, берет данные из тела запроса)
PUT/PATCH - обновление ресурса (изменяет состояние, берет данные из тела запроса)
DELETE - удаление ресурса (изменяет состояние)

Пример плохого API (не REST):
GET /products?action=getAll  -  странный способ получить товары
GET /products?action=delete&id=5  -  удаление через GET (опасно! может быть нажато ссылкой)

Пример хорошего REST API (так правильно):
GET /products  -  явно говорит что это получение данных
DELETE /products/5  -  явно говорит что это удаление


3. АСИНХРОННОСТЬ (Async Programming)

Асинхронное программирование это когда код выполняется не по порядку и не блокирует основное выполнение.
Очень важно для сетевых операций потому что они медленные и могут долго ждать.

Пример синхронного кода (плохо):
const data = fetch(url);  // ждет пока сервер ответит... (может быть 5 секунд!)
// UI зависает на 5 секунд!
showData(data);

Пример асинхронного кода с Promises (лучше):
fetch(url)
    .then(response => response.json())
    .then(data => showData(data));  // выполняется когда данные готовы

Пример асинхронного кода с async/await (лучше всего):
const data = await fetch(url).then(r => r.json());
showData(data);  // выполняется когда данные готовы

Почему это важно: если UI будет ждать загрузки - пользователь не сможет нажать на кнопку, 
скроллить страницу или что-то еще. Асинхронность позволяет UI оставаться отзывчивым.


4. ЛОКАЛЬНОЕ ХРАНИЛИЩЕ (localStorage)

localStorage это браузерное хранилище для сохранения данных на компьютере пользователя.
Данные сохраняются даже после закрытия браузера и остаются на несколько лет.

Как работает:
localStorage.setItem('key', 'value');  // сохранить
const value = localStorage.getItem('key');  // загрузить
localStorage.removeItem('key');  // удалить

Ограничения:
- Примерно 5-10 MB на сайт (зависит от браузера)
- Хранит только строки (поэтому используют JSON.stringify и JSON.parse)
- Одна куча data на user + domain (не синхронизируется между браузерами)
- Нельзя использовать для sensitive данных типа пароль (видно в DevTools)

Применение в проекте:
- Сохранение корзины в localStorage
- При открытии приложения загружаем корзину из localStorage
- Если пользователь закрыл браузер - корзина не потеряется!

Пример:
// При добавлении товара в корзину
let cart = JSON.parse(localStorage.getItem('cart')) || [];
cart.push({ id: 1, product: 'iPhone' });
localStorage.setItem('cart', JSON.stringify(cart));

// При открытии приложения
let savedCart = JSON.parse(localStorage.getItem('cart')) || [];
showCart(savedCart);


5. РАЗДЕЛЕНИЕ ОТВЕТСТВЕННОСТИ (Separation of Concerns)

Это принцип что разные части приложения должны отвечать за разные вещи:

Frontend отвечает за:
- Как выглядит интерфейс (HTML, CSS)
- Как пользователь взаимодействует (клики, ввод)
- Простые валидации (название не пусто, цена число)
- Отображение данных красиво

Backend отвечает за:
- Где хранятся данные (БД)
- Логика обработки данных (комплексные правила)
- Авторизация и безопасность
- Взаимодействие с другими системами

API - это контракт между ними:
Frontend говорит: "Дай мне все товары"
Backend отвечает: "Вот JSON с товарами"

Если разделение нарушено (плохо):
Frontend сохраняет товары напрямую в БД (небезопасно! Пользователь может отправить вредоносные данные)

Если разделение соблюдается (хорошо):
Frontend отправляет данные на сервер
Backend проверяет что данные корректные и сохраняет в БД
Frontend получает подтверждение

ЧАСТЫЕ ОШИБКИ И КАК ИХ ИЗБЕЖАТЬ
════════════════════════════════════════════════════════════════════════════════

1. Забыли запустить сервер
Ошибка: "Cannot reach localhost:3000" или "Network error"
Решение: убедитесь что сервер запущен (npm start в папке server) и работает на порту 3000

2. CORS ошибка при запросе к другому домену
Ошибка: "No 'Access-Control-Allow-Origin' header"
Решение: добавьте app.use(cors()) в server.js (конкретно это уже добавлено в проекте)

3. localStorage содержит старые данные
Проблема: после изменения структуры товара, localStorage содержит старый формат
Решение: очистите браузерные данные через DevTools → Storage → Clear All

4. Асинхронный код не ждет результата
Ошибка: данные undefined потому что код не ждал ответа сервера
Решение: используйте async/await или .then() чтобы ждать результата

5. Модифицирующие операции через GET
Ошибка: добавление товара через GET /api/products?action=add
Решение: используйте POST для создания, DELETE для удаления

6. Забыли JSON.stringify когда сохраняют в localStorage
Ошибка: localStorage содержит "[object Object]" вместо JSON
Решение: всегда используйте JSON.stringify() и JSON.parse()
